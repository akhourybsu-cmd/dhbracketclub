
## Findings from a full sweep

I scanned everything that touches what shows on Home, plus every layer of the cache/update pipeline. Three important facts:

1. **The new Home is in the code and is live in production.** I fetched `https://dhbracketclub.lovable.app/version.json` directly and got `{"buildId":"1776562477014"}` — a real build ID, not "dev". The route `/dashboard` renders `DashboardPage.tsx`, which contains all the new sections: Season Command Strip, Needs Your Attention, Upcoming Events, **League Snapshot** (with podium + progress bar), Drafts, Brackets, Recent Activity. Mobile nav's "Home" tab points at `/dashboard`, and `LandingPage` redirects logged-in users to `/dashboard`. So the source-of-truth Home IS the new one.

2. **Your installed PWA is still booting from a stale cached `index.html`.** That stale HTML predates the bootstrap probe we added, so none of our newer "detect & nuke" logic ever runs. The old service worker keeps serving the old shell, which loads the old JS bundle, which renders the old Home — a closed loop. Every "fix" we deployed only helps users who already loaded the *new* `index.html` at least once.

3. **There are real conflicts in the update layer that make recovery less reliable than it should be:**
   - `vite-plugin-pwa` is configured with `registerType: "autoUpdate"`, but **nothing in `src/` actually imports `virtual:pwa-register`** to register or update the SW. So Workbox's auto-update path is half-wired — the plugin generates a SW but the app never calls `registration.update()`.
   - Our custom `public/sw-push.js` is `importScripts`'d into Workbox's generated SW. On `activate` it now nukes ALL caches — including the precache Workbox just populated. The two strategies fight each other.
   - The Workbox-generated SW intercepts navigations with `NetworkFirst (3s timeout)`. On a flaky mobile connection that times out, Android falls back to the cached old shell, and you're back in the loop.
   - There's no top-level "kill switch" the user can reach without first loading new JS (the Profile button helps, but only if they can get past the old cached shell to find it).

## The fix: kill the old SW from the outside, then simplify

The only thing the *old* installed PWA still fetches reliably is `/sw-push.js` (the SW re-fetches its imported scripts on update). We will use that as the kill vector.

### What we'll change

1. **Turn `public/sw-push.js` into a self-destructing SW.** Replace its contents with a tiny script that, on `install`/`activate`:
   - Calls `self.registration.unregister()`
   - Deletes every cache in `caches.keys()`
   - Calls `clients.claim()` and posts a message telling all open clients to hard-reload with a cache-busted URL
   This breaks the loop on Android the next time the OS pings the SW for updates (typically within 24h, but also on every app launch).

2. **Remove `vite-plugin-pwa` entirely.** It's the source of the stale `index.html` problem, the precache that fights our SW, and the navigation interception. Since you chose "Always fresh", we drop it. Effects:
   - No more service worker generated by the build.
   - No more workbox precache.
   - The app behaves like a normal web app: every visit fetches fresh HTML from the CDN.
   - "Add to Home Screen" still works via a plain `manifest.webmanifest` (we'll add a minimal one so the icon/standalone behavior is preserved on new installs).
   - **Push notifications keep working** — we register `/sw-push.js` ourselves from `src/main.tsx` (after the unregister-everything one-time cleanup runs), and only for the push handlers. No caching, no fetch interception.

3. **Rewrite `/sw-push.js` (post-cleanup version).** After the kill phase ships, we replace it with a *minimal push-only* SW: only `push` and `notificationclick` handlers, no `fetch`, no caching. This makes pushes survive without ever caching content again.
   - We'll do this as a single file that handles both phases via a version flag, so we don't have to deploy twice.

4. **Strengthen the in-HTML bootstrap probe.** Keep the existing probe in `index.html` but also: on every load, proactively call `navigator.serviceWorker.getRegistrations()` and unregister anything whose script URL is NOT `/sw-push.js`. This evicts the old Workbox-generated SW (`/sw.js` or `/registerSW.js`) immediately on first new-HTML load.

5. **Drop the now-redundant pieces** so future-you doesn't trip on them:
   - Remove `VitePWA(...)` from `vite.config.ts` and the `vite-plugin-pwa` dependency.
   - Remove `/// <reference types="vite-plugin-pwa/react" />` from `src/vite-env.d.ts`.
   - Keep `useAppUpdate.ts`, `forceUpdate.ts`, the Profile "Force Update" button, `version.json`, and the bootstrap probe — they all still work and give us multiple recovery layers.

### Files touched
- `public/sw-push.js` — rewritten as self-destruct + minimal push handler
- `public/manifest.webmanifest` — new, minimal (keeps installability)
- `index.html` — link the new manifest, beef up bootstrap probe to evict non-`/sw-push.js` SWs, register `/sw-push.js` for push only
- `src/main.tsx` — remove any PWA-register references; ensure cleanup runs before render
- `vite.config.ts` — remove `VitePWA` plugin entirely
- `package.json` — remove `vite-plugin-pwa`
- `src/vite-env.d.ts` — remove pwa types reference
- `public/version.json` — bumped (build also rewrites it)

### Why this works on your stuck install
The next time Android wakes the old SW (which it does on every app open and periodically in background), it will refetch its imported `sw-push.js`. The new contents will unregister the SW, wipe all caches, and reload every open client. From that point on, there is no SW intercepting navigations, so `index.html` always comes fresh from the CDN, and you see whatever the latest deploy is — every time.
